# 星载可靠推理项目与论文最终说明

## 1. 论文定位

建议论文题目继续使用：

> 面向星载通用处理平台的神经网络可靠推理与资源约束软件保护方法研究

论文主线不是“重新做一个推理框架”，也不是“提出 DMR/range guard 这类传统容错方法”。当前最稳的主线是：

> 面向星载遥感推理任务，在时延和峰值内存受限条件下，评估运行时对象故障造成的任务后果，并自动选择低开销的软件保护配置，最终生成可执行的受保护推理程序。

本文只声称给定软件故障模型下的相对可靠性改进，不声称真实辐照等效，不声称当前 Windows 平台等价于星载认证平台。

## 3. 最终采用的方法

### 3.1 任务后果感知的运行时对象风险评估

传统层敏感性分析容易只看数值误差或层级准确率。本文把故障影响直接映射到任务后果：

- 分类任务：原本预测正确的样本在故障后分类错误，或置信度显著异常。
- 检测任务：漏检、误检、类别改变、定位偏差、执行错误。

风险评分由故障注入估计的关键失效概率、严重度和 activation-byte 暴露代理共同构成。该部分支撑“哪些运行时对象值得保护”的问题抽象。

### 3.2 Confidence-Robust Bounded-Loss Memory ILP

该方法是当前最适合作为算法创新点的部分。

基本思想：

1. 不直接使用单次实验均值风险收益，而使用由正式故障日志估计的置信稳健收益。
2. 先求给定时延和内存预算下的稳健风险收益最大方案。
3. 允许一个可解释的风险收益损失上限，例如 `5%`。
4. 在该损失约束内最小化峰值额外保护内存。

这个方法解决的问题是：星载受限平台上，不应该为了极小的风险收益差异选择大缓冲 DMR。实验中代表预算点风险收益保持约 `97.39%`，峰值额外保护内存降低 `100%`。

### 3.3 Task-Utility Consistency Guard

普通 `task_output_guard` 只检查输出是否合法，例如 NaN/Inf、置信度越界、类别越界、box 越界等。增强后的 Task-Utility Consistency Guard 增加任务效用一致性检查：

- 检测目标数量突变。
- 高置信目标数量突变。
- 检测置信度总量突变。
- 类别直方图异常变化。
- box 面积分布异常。

在 YOLOv10n + DIOR 的 10000 个 runtime fault event 重放中：

- 普通 task output guard 关键失效降低率：`50.52%`。
- Task-Utility Consistency Guard 关键失效降低率：`97.21%`。
- clean false positive：`0`。
- guard 触发率从 `1.46%` 增加到 `2.80%`。

该机制可以作为检测任务特色保护方法写入论文。

## 4. 未采用或只作为消融的方法

### 4.1 Guard-Gated DMR

结果：只作为消融，不作为主方法。

该方法通过轻量 guard 触发按需 DMR。实验显示时延降低约 `76.99%`，但恢复收益保持率只有约 `62.16%`，低于主方法门槛 `80%`。因此不能作为核心方法，否则会削弱可靠性论证。

### 4.2 Adaptive Quantile Range Guard

结果：不采用。

这部分已经做了真实重放验证，不是没有实验。

实验设置：

- ResNet50 top-8 高风险节点。
- 每节点 32 个故障事件。
- 比较 `minmax`、`q99.9`、`q99.5`、`q99`。
- clean holdout 总计 `16384` 次策略检查。

结果：

| 策略 | 关键故障数 | 检出的关键故障数 | 关键覆盖率 | clean false positive |
| --- | ---: | ---: | ---: | ---: |
| minmax | 7 | 0 | 0.0 | 0.0 |
| q99.9 | 7 | 0 | 0.0 | 0.0 |
| q99.5 | 7 | 0 | 0.0 | 0.0 |
| q99 | 7 | 0 | 0.0 | 0.0 |

为什么效果不好：

1. Range guard 检测的是“激活值是否超出正常范围”。这类方法擅长发现 NaN、Inf、极大值、明显越界值。
2. 本次重放中的关键分类错误并没有表现为全局范围越界。也就是说，模型输出错了，但被注入节点的激活值仍落在正常分布范围内。
3. Quantile 阈值确实比 minmax 更敏感，实验中 q99/q99.5/q99.9 检出了一些非关键异常，但没有检出关键故障。
4. 对论文目标而言，保护机制关注的是“减少任务关键失效”，不是单纯提高异常报警数。多报非关键异常只会增加重执行开销，不能支撑主方法。

因此 Adaptive Quantile Range Guard 当前不进入论文主线。它可以作为负结果说明：单纯收紧激活范围阈值不一定能覆盖任务级关键失效。

## 5. 正式实验结果

| 模型 | 实验 | 事件数 | 未保护失效 | 方法 | 保护后失效 | 下降比例 |
| --- | --- | ---: | ---: | --- | ---: | ---: |
| ResNet50 + EuroSAT | 多 seed runtime fault | 30000 | 555 | multi-mode ILP + range guard + DMR | 155 | 72.07% |
| YOLOv10n + DIOR | activation-prior runtime fault | 10000 | 287 | ILP-DMR | 240 | 16.38% |
| YOLOv10n + DIOR | task output guard | 10000 | 287 | task output guard | 142 | 50.52% |
| YOLOv10n + DIOR | task utility guard | 10000 | 287 | Task-Utility Consistency Guard | 8 | 97.21% |

机制筛选最终结论：

| 候选方法 | 结论 | 写法 |
| --- | --- | --- |
| Guard-Gated DMR | `keep_as_ablation` | 可作为消融，不作为创新 |
| Adaptive Quantile Range Guard | `reject` | 作为负结果或不写 |
| Task-Utility Consistency Guard | `adopt_as_main` | 检测任务特色保护机制 |
| Confidence-Robust Bounded-Loss Memory ILP | `adopt_as_main` | 资源约束优化核心方法 |

## 6. 论文创新点建议

建议最终写三条创新点：

1. **任务后果感知的运行时对象脆弱性评估方法**  
   将运行时对象故障与分类错误、检测漏检、误检、定位退化和执行错误关联，形成任务级风险排序和稳定性分析。

2. **置信稳健的风险损失受限低内存保护配置优化方法**  
   使用故障注入统计的置信稳健收益，在允许有限风险收益损失的条件下最小化峰值额外保护内存，适配星载受限平台。

3. **面向遥感检测任务的任务效用一致性保护机制与可执行落地**  
   针对 YOLO 检测输出定义任务效用一致性检查，并通过 `ProtectionPlan`、SPK、Runtime 和 Generated C 验证保护方案可执行。

不要把以下内容单独写成创新：

- DMR/TMR。
- range guard。
- 故障注入工具。
- 中间层敏感性分析。
- 通用推理框架或 ONNX 转译。

## 7. 论文结构建议

### 第 1 章 绪论

写星载遥感在轨处理背景、软错误风险、资源受限问题和全量冗余代价过高的问题。

### 第 2 章 相关工作

覆盖星载 AI、DNN 软错误、故障注入、软件容错、选择性保护、推理部署。重点说明本文不重复加速器容错和传统 DMR，而是做任务后果、低内存优化和可执行落地。

### 第 3 章 任务后果感知的风险评估方法

定义故障模型、运行时对象、任务失效指标、风险评分、置信区间和排序稳定性。

### 第 4 章 资源约束保护配置优化方法

写保护候选、开销模型、普通 ILP、置信稳健收益、风险损失受限低内存优化，以及和 random、Top-k、greedy、普通 ILP 的对比。

### 第 5 章 实验与分析

写 ResNet50 和 YOLOv10n 的完整实验：

- 模型和数据。
- 故障注入设置。
- 风险排序。
- 保护效果。
- 机制筛选。
- Task-Utility Consistency Guard。
- 工程指标和 Generated C smoke。

### 第 6 章 总结与展望

总结方法有效性和限制。明确 D2000/RISC-V 尚未完成，后续作为实机迁移验证。
